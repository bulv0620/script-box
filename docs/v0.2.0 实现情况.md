# Script Box v0.2.0 实现情况

## 版本定位

v0.2.0 是 Script Box 的 Job 扩展版本。

本版本在 v0.1.0 的任务管理、运行日志、状态同步和打包能力基础上，重点完成 Job 管理能力升级：

- 支持内置 Job 和用户导入 Job 双来源
- 支持导入 zip 格式 Job 包
- 支持 Job 自带 `package.json`
- 支持导入后自动安装依赖
- 支持重新安装用户 Job 依赖
- 支持删除用户 Job
- 支持打开用户 Job 目录
- 支持导入过程进度和错误提示
- 移除内置 demo Job

## 技术栈

技术栈延续 v0.1.0：

- Electron
- Vue 3
- TypeScript
- Pinia
- Vuetify
- SQLite, via `better-sqlite3`
- Electron Vite
- electron-builder

新增或强化的能力主要集中在：

- Electron 主进程文件系统能力
- zip 解压和校验
- 外部 Node/npm 环境检测
- Job 级别依赖安装
- Renderer Job 管理弹窗

## Job 来源

v0.2.0 将 Job 来源拆成两类。

### 内置 Job

内置 Job 仍放在项目源码中：

```text
src/jobs/
```

打包后随应用发布。

当前保留的内置 Job：

- `http-job`

已移除：

- `demo-job`

内置 Job 是只读 Job：

- 不允许删除
- 不支持重新安装依赖
- 随应用版本发布和更新

### 用户 Job

用户导入的 Job 存放在 Electron `userData` 目录：

```text
userData/jobs/
```

每个 Job 独立一个目录：

```text
userData/jobs/
  auto-shelf-job/
    manifest.js
    index.js
    package.json
    package-lock.json
    node_modules/
```

用户 Job 支持：

- 导入
- 删除
- 重新安装依赖
- 打开所在目录查看

## Job 包格式

导入 Job 使用 zip 包。

zip 根目录可以直接包含 Job 文件，也可以只包含一个 Job 目录。

必须文件：

```text
manifest.js
index.js
package.json
```

可选文件：

```text
package-lock.json
README.md
assets/
```

`package.json` 必须包含：

```json
{
  "type": "module"
}
```

原因是当前 Job 使用 ESM 动态 `import()` 加载。

## Job Manifest

`manifest.js` 需要默认导出 Job 元信息：

```js
export default {
  jobId: 'auto-shelf-job',
  jobName: '自动上下架 Job',
  version: '1.0.0',
  description: '根据接口数据自动执行商品上下架',
  config: []
}
```

导入时会校验：

- `manifest.js` 存在
- `index.js` 存在
- `package.json` 存在
- `manifest.js` 默认导出必须是对象
- `jobId` 不能为空
- `jobName` 不能为空
- `version` 不能为空
- `config` 必须是数组
- `package.json` 必须是合法 JSON
- `package.json` 必须包含 `"type": "module"`
- `jobId` 不能和已有 Job 冲突

加载后应用会补充运行时字段：

- `source`
- `rootDir`

用于区分内置 Job 和用户 Job，并支持删除、安装依赖等管理动作。

## Job 运行接口

`index.js` 需要默认导出 Job 实现：

```js
import manifest from './manifest.js'

export default {
  manifest,

  async run(config, context) {
    context.logger.info('开始执行')
  }
}
```

运行机制仍沿用 v0.1.0 的 Runner：

```text
Task
  -> RunnerManager
  -> JobManager.get(jobId)
  -> 动态 import Job
  -> job.run(config, context)
```

因此导入后的用户 Job 可以直接获得现有能力：

- 任务运行
- 任务停止
- 实时日志
- 历史日志
- 运行记录
- 状态同步

## 依赖安装

用户 Job 可以在自己的 `package.json` 中声明依赖。

示例：

```json
{
  "name": "auto-shelf-job",
  "version": "1.0.0",
  "type": "module",
  "dependencies": {
    "axios": "^1.7.0"
  }
}
```

导入成功后，应用会在该 Job 目录执行依赖安装：

如果存在 `package-lock.json`：

```bash
npm ci --omit=dev
```

如果不存在 `package-lock.json`：

```bash
npm install --omit=dev
```

Windows 下通过 `cmd.exe /d /s /c` 执行 npm，避免直接 spawn `npm` 在部分环境下出现 `spawn EINVAL`。

安装过程中的输出会通过 Job 导入进度推送到前端。

## Node/npm 环境检测

v0.2.0 仍然依赖用户本机 Node/npm 来安装外部 Job 依赖。

用户点击“导入 Job”时，会先检测：

```bash
node -v
npm -v
```

如果本机没有可用的 Node 或 npm：

- 页面展示错误提示
- 导入流程停止
- 自动打开 Node.js 官网

下载地址：

```text
https://nodejs.org/
```

注意：Electron 自身的 Node 运行时可以执行应用代码，但不能直接等价为完整的用户本机 Node/npm 环境。当前依赖安装仍使用系统命令中的 npm。

## 依赖解析方式

用户 Job 仍通过动态 `import()` 加载。

当用户 Job 位于：

```text
userData/jobs/auto-shelf-job/index.js
```

并在 `index.js` 中写入：

```js
import axios from 'axios'
```

Node 会从 Job 自己目录开始向上查找依赖，优先使用：

```text
userData/jobs/auto-shelf-job/node_modules/
```

因此用户 Job 应该把运行依赖声明在自己的 `package.json` 中，不建议依赖 Script Box 应用自身的 `node_modules`。

应用提供给 Job 的稳定能力应通过 `context` 注入。

## Job 管理 UI

v0.2.0 新增 Job 管理弹窗。

入口位置：

```text
顶部操作区 -> 新增任务右侧 -> Job 管理
```

当前支持：

- 查看内置 Job
- 查看用户 Job
- 导入 Job
- 删除用户 Job
- 重新安装用户 Job 依赖
- 打开用户 Job 目录
- 查看导入和安装进度
- 查看失败原因

删除用户 Job 时，如果该 Job 已有关联任务，会阻止删除并在页面提示错误。

## 导入流程

当前导入流程：

1. 点击导入 Job
2. 检测本机 Node/npm 环境
3. 选择 zip 文件
4. 解压到临时目录
5. 校验 Job 结构
6. 读取并校验 `manifest.js`
7. 读取并校验 `package.json`
8. 检查 `jobId` 是否已存在
9. 复制到 `userData/jobs/<jobId>`
10. 安装依赖
11. 动态加载导入后的 Job
12. 刷新 Job 列表
13. 返回导入成功结果

导入失败时：

- 会清理临时目录
- 页面会展示失败原因
- 已有 Job 列表不会被整个异常拖垮

如果某个用户 Job 损坏，JobManager 会跳过该 Job 并继续加载其他 Job，避免一个坏包导致内置 Job 或其他用户 Job 全部消失。

## 已移除能力

v0.2.0 期间曾规划过导出 Job，但当前已移除。

原因：

- 当前主要需求是导入和本地管理
- 导出 Job 对当前使用场景收益不高
- 可以降低 UI 和 IPC 复杂度

当前不再提供：

- 导出 Job 按钮
- `job:export` IPC
- 导出 zip 相关主进程逻辑

## IPC

v0.2.0 新增或扩展 Job 相关 IPC：

```text
job:list
job:get
job:import
job:delete
job:installDeps
job:openUserJobsDir
job:progress
```

其中：

- `job:import` 用于导入 zip Job 包
- `job:delete` 只允许删除用户 Job
- `job:installDeps` 只允许对用户 Job 执行
- `job:openUserJobsDir` 打开用户 Job 目录
- `job:progress` 推送导入和安装过程

## 安全说明

导入外部 Job 本质上是在本地运行用户脚本。

外部 Job 拥有 Node.js 能力，可能访问：

- 本机文件
- 网络请求
- 环境变量
- 本机可用命令

当前版本没有权限隔离和沙盒机制。

建议用户只导入可信来源的 Job。

## 当前限制

### 依赖安装依赖本机 npm

当前应用没有内置 npm。

如果用户电脑没有安装 Node/npm，外部 Job 导入会被阻止，并引导用户安装 Node.js。

### 停止机制仍是协作式

任务停止仍依赖 `AbortSignal`。

Job 需要主动检查：

```js
if (context.signal.aborted) {
  return
}
```

如果 Job 执行长时间同步阻塞逻辑，当前版本无法强制中断。

### Job 仍在主进程内执行

当前 Job 仍运行在 Electron 主进程内。

风险：

- Job 异常可能影响主进程稳定性
- 无法进程级强杀
- 暂无 CPU 和内存限制
- 暂无权限模型

### 暂无 Job 更新机制

当前如果导入同名 `jobId`，会提示已存在。

暂未支持：

- 覆盖安装
- 版本升级
- 回滚
- 禁用 Job
- Job 市场

## v0.2.0 总结

v0.2.0 已经把 Script Box 从“只能运行内置脚本”的工具，升级为“可以导入和管理用户 Job”的本地脚本任务平台。

当前版本已经具备外部 Job 的基础闭环：

- zip 分发
- 结构校验
- 自动安装依赖
- 动态加载
- 创建任务
- 运行任务
- 停止任务
- 日志监控
- 错误提示

## 下一步升级规划

下一阶段建议围绕“执行隔离、稳定性和 Job 生命周期管理”继续推进。

推荐版本方向：v0.3.0。

### 子进程执行

建议将 Job 从主进程内执行迁移到独立子进程：

```text
Main Process
  -> RunnerManager
  -> child_process fork/spawn
  -> Job Worker
```

收益：

- 可以强制 kill
- Job 崩溃不拖垮主进程
- stdout/stderr 可以接入日志
- 每个 Job 的运行环境更清晰

### Job 生命周期管理

建议后续支持：

- 覆盖更新用户 Job
- 禁用/启用 Job
- 显示 Job 使用中的任务数量
- 删除 Job 前展示关联任务
- Job 版本信息和更新时间
- 重新导入同名 Job 时做版本比较

### 依赖安装增强

建议后续支持：

- npm registry 配置
- 安装超时
- 安装失败重试
- 取消安装
- 更完整的安装日志查看
- 离线依赖包方案

### 安全和权限

建议后续增加：

- 导入前风险提示
- manifest 权限声明
- 用户确认权限
- 文件访问范围限制
- 网络访问声明
- 子进程沙盒或更严格执行环境

### Job 开发体验

建议后续补充：

- manifest 类型定义
- 示例 Job 文档

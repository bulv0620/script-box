# Script Box v0.1.0 实现情况

## 版本定位

v0.1.0 是 Script Box 的第一个可用 MVP 版本。

本版本目标是完成桌面端脚本任务管理的基础闭环：

- 自动发现 Job
- 创建和管理 Task
- 执行和停止 Task
- 查看实时日志
- 查看历史运行记录
- 使用 SQLite 保存任务和运行记录
- 支持 Windows 安装包和免安装包打包

## 技术栈

- Electron
- Vue 3
- TypeScript
- Pinia
- Vuetify
- SQLite, via `better-sqlite3`
- Electron Vite
- electron-builder

## 已实现模块

### Electron 主进程

主进程已拆分为以下模块：

```text
src/main/
  db/          SQLite 初始化与迁移
  ipc/         IPC 接口注册
  job/         Job 自动发现和动态加载
  log/         日志文件写入、读取和实时推送
  runner/      Task 执行、停止、状态维护
  task/        Task CRUD 和状态更新
  types.ts     主进程共享类型
```

窗口使用无边框模式：

- 自定义标题栏
- 最小化、最大化/还原、关闭按钮
- 标题栏拖拽区域
- 应用图标

### Renderer 前端

前端已实现：

- 任务总览
- 任务统计
- 任务列表
- 新增任务
- 编辑任务
- 删除任务
- 运行任务
- 停止任务
- 日志弹窗
- 实时日志追加
- 执行记录选择
- 日志搜索
- 日志复制
- 日志下载

页面布局当前采用：

- 固定顶部标题栏
- 主内容固定窗口高度
- 底部任务列表区域独立滚动
- 新增/编辑任务弹窗的 header 和 footer 固定，中间表单独立滚动

### Job 系统

当前 Job 放在：

```text
src/jobs/
```

每个 Job 目录包含：

```text
manifest.js
index.js
```

`manifest.js` 定义 Job 元信息和配置字段。

`index.js` 实现标准接口：

```js
export default {
  manifest,

  async run(config, context) {
  }
}
```

Job 通过主进程动态 `import()` 加载。

当前内置示例：

- `http-job`

### Task 系统

Task 是 Job 的用户配置实例。

已支持：

- 创建
- 编辑
- 删除
- 查询
- 运行
- 停止

Task 状态：

- `idle`
- `running`
- `success`
- `failed`
- `stopped`

任务列表当前按 `created_at DESC` 排序。

运行、停止、完成、失败不会改变任务在列表中的顺序。

### Runner

Runner 已实现：

- 防止同一个任务重复运行
- 每次运行生成 RunRecord
- 每次运行生成独立日志文件
- 使用 `AbortController` 停止任务
- 根据 Job manifest 的 `timeout` 做超时控制
- 执行结束后自动更新 Task 状态
- 执行状态变化通过 IPC 推送到前端

当前停止机制是协作式停止。

Job 需要定期检查：

```js
context.signal.aborted
```

如果 Job 执行阻塞同步代码或死循环，当前版本无法强制中断。

### 日志系统

日志文件保存在 Electron `userData` 目录下：

```text
logs/
```

每次运行生成一个日志文件。

日志格式：

```text
[2026-06-17 10:00:00] INFO 开始执行
```

已支持：

- 实时写入日志文件
- 实时推送到 Renderer
- 历史日志读取
- 日志复制
- 日志下载
- 日志搜索

### RunRecord 保留策略

每个任务只保留最近 50 条运行记录。

执行结束后会自动清理：

- 第 51 条及更旧的 `run_record`
- 对应日志文件

日志文件删除做了路径校验，只会删除日志目录内文件。

### 数据库

数据库使用 SQLite，存储在 Electron `userData` 目录：

```text
database/script-box.sqlite
```

当前表：

- `task`
- `run_record`

数据库迁移在应用启动时自动执行。

## 当前限制

### Job 来源限制

当前只支持内置 Job。

生产环境 Job 路径指向应用内的：

```text
src/jobs/
```

打包后手工向外部目录新增 Job 不会自动生效。

### Job 依赖限制

内置 Job 可以使用应用自身的 `node_modules`。

当前尚未支持外部 Job 自带 `package.json`、自动安装依赖、导入和导出。

### 停止机制限制

当前停止机制依赖 `AbortSignal`。

如果 Job 没有检查 `context.signal.aborted`，或者执行了长时间阻塞同步代码，停止不会及时生效。

### 安全隔离限制

当前 Job 在 Electron 主进程内执行。

这意味着：

- Job 能访问 Node 能力
- Job 出错可能影响主进程稳定性
- 暂无权限隔离
- 暂无进程级强制停止

## v0.1.0 总结

v0.1.0 已经完成脚本任务管理工具的基础闭环。

当前版本适合作为本地个人脚本任务管理器的基础版本，也为后续扩展外部 Job、脚本导入导出、依赖安装、任务调度和更强隔离打好了结构基础。

## 下一步升级规划

### 升级目标

下一阶段建议围绕“可扩展 Job 系统”推进。

核心目标：

- 支持外部 Job 导入
- 支持外部 Job 导出
- 支持 Job 自带 `package.json`
- 导入后自动安装依赖
- 保留现有运行、停止、日志监控能力
- 为后续脚本市场、更新和隔离执行打基础

### 推荐版本方向

建议下一阶段版本定义为 v0.2.0。

v0.2.0 的重点不是大改 UI，而是把 Job 从“只读内置脚本”升级为“可导入、可管理的用户脚本”。

### Job 来源改造

当前 Job 只有内置来源：

```text
src/jobs/
```

建议升级为双来源：

```text
内置 Jobs
  app 内置，只读，随应用发布

用户 Jobs
  userData/jobs，可导入、导出、删除、更新
```

推荐目录：

```text
userData/
  jobs/
    email-job/
      manifest.js
      index.js
      package.json
      package-lock.json
      node_modules/
```

### Job 数据模型升级

建议 Job manifest 在应用内部加载后补充运行时字段：

```ts
interface RuntimeJobManifest {
  jobId: string
  jobName: string
  description?: string
  version: string
  timeout?: number
  config: ConfigField[]
  source: 'builtin' | 'user'
  rootDir: string
}
```

其中：

- `source` 用于区分内置 Job 和用户导入 Job
- `rootDir` 用于导出、删除、依赖安装和运行定位

内置 Job：

- 不允许删除
- 不允许覆盖更新，除非后续做官方更新机制

用户 Job：

- 允许导入
- 允许导出
- 允许删除
- 允许更新

### 导入脚本设计

导入格式建议使用 zip。

zip 根目录必须包含：

```text
manifest.js
index.js
package.json
```

可选：

```text
package-lock.json
README.md
assets/
```

导入流程：

1. 用户选择 zip
2. 解压到临时目录
3. 校验必须文件
4. 动态加载 `manifest.js`
5. 校验 `jobId`
6. 检查 Job 是否已存在
7. 复制到 `userData/jobs/<jobId>`
8. 在 Job 目录执行依赖安装
9. 刷新 Job 列表
10. 返回导入结果

建议导入前校验：

- `manifest.js` 必须能加载
- `jobId` 不能为空
- `jobName` 不能为空
- `version` 不能为空
- `config` 必须是数组
- `index.js` 必须导出 `run`
- `package.json` 必须存在
- `package.json` 建议包含 `"type": "module"`

### 依赖安装策略

用户 Job 依赖必须声明在自己的 `package.json`。

示例：

```json
{
  "name": "email-job",
  "version": "1.0.0",
  "type": "module",
  "dependencies": {
    "nodemailer": "^6.9.0"
  }
}
```

导入后在 Job 根目录执行：

```bash
npm install --omit=dev
```

或如果存在 lockfile：

```bash
npm ci --omit=dev
```

建议 v0.2.0 先使用 npm。

后续可以支持：

- pnpm
- yarn
- 离线依赖包
- 依赖安装日志
- 依赖安装失败重试

### 外部 Job 的依赖解析

外部 Job 仍使用动态 `import()` 加载。

当 Job 位于：

```text
userData/jobs/email-job/index.js
```

Job 内部导入：

```js
import nodemailer from 'nodemailer'
```

Node 会优先查找：

```text
userData/jobs/email-job/node_modules
```

因此外部 Job 可以使用自己安装的依赖。

建议规则：

```text
外部 Job 必须声明并安装自己的运行依赖。
不要依赖 Script Box 应用自身 dependencies。
```

应用提供给 Job 的稳定能力应通过 `context` 注入，而不是要求 Job 直接 import 应用内部模块。

### 运行、停止和日志监控

导入 Job 后，运行机制可以继续沿用现有 Runner。

执行流程：

```text
RunnerManager
  -> JobManager.get(jobId)
  -> import(userData/jobs/<jobId>/index.js)
  -> job.run(config, { logger, signal })
```

日志机制不需要重做：

```text
Job
  -> context.logger.info()
  -> LogManager 写入日志文件
  -> IPC log:event
  -> Renderer 实时展示
```

停止机制也可以继续使用：

```text
task:stop
  -> AbortController.abort()
  -> context.signal.aborted
  -> Job 主动退出
```

需要在外部 Job 开发规范中明确：

```js
if (context.signal.aborted) {
  return
}
```

长任务必须分步骤执行，并定期检查 `signal.aborted`。

### 导出脚本设计

导出用户 Job 时，将 Job 目录打包成 zip。

建议包含：

```text
manifest.js
index.js
package.json
package-lock.json
README.md
assets/
```

不建议默认导出：

```text
node_modules/
```

原因：

- 文件体积大
- 平台相关依赖可能不可移植
- 安全和兼容性更复杂

可以后续提供高级选项：

```text
导出时包含 node_modules
```


### 更强隔离的后续方向

当前 Runner 是 in-process 执行。

未来如果需要更强停止能力和稳定性，建议迁移为：

```text
child_process
```

或：

```text
worker_threads
```

推荐优先考虑 `child_process`：

- 可以直接 kill
- 崩溃不会拖垮主进程
- 每个 Job 依赖解析更清晰
- stdout/stderr 可接入日志

但这会带来更多 IPC 和运行协议设计，建议放到 v0.3.0 或之后。
